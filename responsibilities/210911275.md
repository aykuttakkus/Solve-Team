### What I Did

#### 1. Dataset Selection and Problem Definition

I identified the Breast Cancer Wisconsin (Diagnostic) dataset as the appropriate foundation for this project. The dataset presents a binary classification task (Malignant vs. Benign) with 569 samples and 30 continuous numerical features derived from digitized images of fine needle aspirates.

The rationale for selection was methodological: the dataset is clean enough to isolate algorithmic behavior (no missing values, no categorical encoding needed), small enough to demonstrate overfitting with an MLP, and linearly separable enough to make Logistic Regression a meaningful baseline — directly connecting to Week 2 material.

#### 2. Exploratory Data Analysis

- Verified dataset dimensions: 569 samples × 30 features
- Confirmed zero missing values across all features
- Analyzed class distribution: 357 Benign (62.7%) vs. 212 Malignant (37.3%)
- Established that the naive baseline (always predicting Benign) yields 62.7% accuracy — used as the floor against which all models are evaluated
- Computed descriptive statistics and identified extreme feature scale differences (e.g., `mean area` ranging up to 2501.0 vs. `mean fractal dimension` at 0.0017)

#### 3. Multicollinearity and Target Correlation Analysis

Produced a correlation heatmap for the top features correlated with the target variable. Identified feature pairs with correlations exceeding 0.99 (e.g., `worst radius` and `worst perimeter`), which directly motivated the use of L2 regularization in Step 4 — high multicollinearity causes unconstrained MLP weights to become unstable.

**Output:** `outputs/figures/step1/top_feature_correlation_heatmap.png`

#### 4. Preprocessing Pipeline — StandardScaler with Strict Data Leakage Prevention

Applied Z-Score Normalization (StandardScaler) exclusively to the training set:

- `StandardScaler.fit_transform()` applied **only** to `X_train`
- `.transform()` applied to `X_val` and `X_test` using training statistics

This prevents data leakage: the model never "sees" the scale of validation or test data during training. The scale difference between raw and normalized features is documented in `outputs/tables/scaling_check.csv`.

#### 5. Stratified Train / Validation / Test Split

Implemented a stratified split (60% train / 20% validation / 20% test) using a fixed random seed (42) to ensure reproducibility across all notebooks.

Stratification was necessary because the 63/37 class imbalance would cause random splits to produce unrepresentative subsets. The split summary is saved in `outputs/tables/split_summary.csv`.

The shared split logic is encapsulated in `data_utils.py` → `create_stratified_splits()`, which is imported identically by all five notebooks, guaranteeing fair model comparison.

---

### Technical Decisions and Justifications

| Decision | Justification |
|---|---|
| StandardScaler over MinMaxScaler | Gradient-based optimizers (SGD, Adam) converge faster with zero-mean unit-variance inputs; MinMaxScaler is sensitive to outliers |
| Stratified split | Class imbalance (63/37) makes random splits unreliable; stratification preserves the ratio in every subset |
| 60/20/20 split ratio | Gives sufficient training data for MLP while maintaining a held-out test set untouched until Step 5 |
| Fixed seed = 42 | Reproducibility: every team member and every notebook produces identical splits |
| Validation set separate from test | Hyperparameter tuning (Step 4) uses validation; final reported metrics use test — prevents optimistic bias |
